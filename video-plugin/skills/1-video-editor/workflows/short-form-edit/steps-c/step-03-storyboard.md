### 2. Build Storyboard (Per Video)

For each video `sf-{NN}`:

#### A. Load Inputs

- **Clipped refined transcript (PREFERRED):** `short-form/analysis/sf-{NN}/clipped-refined-transcript.json`
  - Timestamps are **native to the clipped video timeline** — no offset math needed
  - This is the default source when available (generated by step-02 section C2)
- **Legacy fallback:** `short-form/analysis/sf-{NN}/refined-transcript.json` + keep-segments offset math
  - Only use if `clipped-refined-transcript.json` is not available
  - Fall back further to `short-form/analysis/sf-{NN}/transcript.json` if refined version also missing
- Keep segments: `short-form/clips/sf-{NN}-keep-segments.json` (needed for legacy offset math only)
- Creative plan: `short-form/scripts/sf-{NN}-script.md` → conceptual storyboard section
- **Total duration:** Get from `ffprobe` on the clipped proxy rather than summing keep-segment durations:
  ```bash
  ffprobe -v error -show_entries format=duration -of csv=p=0 \
    "{project_folder}/{project-slug}/video-editor/short-form/clips/sf-{NN}-cleaned.mp4"
  ```

#### B. Map Conceptual Beats to Real Timestamps

For each beat in the conceptual storyboard:
1. Find the matching words in the transcript using the beat's description
2. Get the word-level timestamps from the transcript

**When using `clipped-refined-transcript.json` (preferred):**
3. Calculate frame numbers directly: `frame = Math.round(timestamp_seconds * fps)`
   - No offset math needed — timestamps are already native to the clipped video timeline
4. Assign `startFrame` and `durationInFrames` for each segment

**Legacy fallback (pre-clip `refined-transcript.json` only):**
3. Calculate frame numbers: `frame = Math.round((timestamp_seconds - clip_offset) * fps)`
   - WARNING: This approach accumulates drift across multiple gap removals and is less accurate
4. Assign `startFrame` and `durationInFrames` for each segment

#### C. Apply Production Style Guide

Before assigning patterns, apply the production style guide constraints:

1. **Layout sequencing**: Start with V5 split-screen (50/50) — recognizable visual top + speaker bottom, alternate with full-frame, end with full-frame speaker for CTA
2. **Cut rhythm**: Target 2–3 cuts in first 3s (hook-fast-then-regular), ~2s intervals in body, single CTA shot
3. **Zoom factors**: V1 speaker = 1.03, V2 emphasis = 1.08 (NOT the old values of 1.08/1.25)
4. **Caption style**: Platform-native — zero burned-in captions in Remotion. SRT generated post-render for YouTube; Instagram/TikTok use platform auto-captions.
5. **Transitions**: Hard-cuts only — do NOT plan dissolves, wipes, or any blended transitions
6. **Split-screen**: 50/50 ratio (content top, speaker bottom), no visible divider (transparent default)

#### C1b. Select Background Music Track

Select the most appropriate background music track from `{audioLibraryData}` for this video:

1. **Determine the video's mood/tone** from the script — common moods: techy, inspiring, energetic, chill, professional, bold
2. **Match against `audio-library.yaml` mood tags** — each track has `mood: [tag1, tag2]` and `energy: low|medium|high|very-high`
3. **Selection heuristics:**
   - Punchy (15–20s) videos → prefer `medium-high` or `high` energy
   - Standard (30–45s) videos → prefer `medium` to `high` energy
   - Deep (50–60s) videos → prefer `medium` energy
   - Tech demo / developer content → prefer `techy` mood tags
   - Inspirational / motivational content → prefer `inspiring`, `uplifting` mood tags
4. **Record in storyboard metadata:**
   - `Background track`: filename from the catalog (e.g., `bg-corporate-tech-01.mp3`)
   - `BPM`: from the catalog entry
   - `Mood match`: which mood tags matched
5. **Track source path:** `{project-root}/example-account-brand-plugin/context/brand/brand-assets/{file}` (the `file` field in `audio-library.yaml` is relative to `brand-assets/`)
6. **SFX source path:** `{project-root}/example-account-brand-plugin/context/brand/brand-assets/sfx/click-soft-01.wav`

This selection feeds into step-04's audio mix (Section 7), where the music + click SFX are pre-mixed into a single MP3 for Remotion.

#### C2. Identify Hero Keyword Visual

Before assigning segment patterns, identify the video's **hero keyword visual** — the single recognizable image that opens the video in the V5 split-screen top half (see P4a in pacing rules).

1. **Read the script's Hook section** — identify the dominant tool, platform, or concept being introduced
2. **Map hero keyword to visual asset type:**
   - Named tool/platform (e.g., "Claude", "VS Code") → logo MG via `fetch-logo.ts` (fetched in step 4, section 5b)
   - Code/dev topic → IDE or terminal interface MG (Hera prompt-only)
   - AI tool → tool logo MG OR chat interface MG
   - Comparison → side-by-side logos (each fetched via `fetch-logo.ts`, composed in step 4)
3. **Record in storyboard metadata:**
   - `heroKeyword`: the identified keyword (e.g., "Claude Code")
   - `heroVisualType`: `logo` | `ui-interface` | `comparison-logos`
   - `heroLogoName`: tool name for `fetch-logo.ts` (if applicable)
4. **First V5 segment's overlay MUST use this hero keyword visual** — this is non-negotiable per P4a
5. **Add hero keyword MG to MG Requests table** with priority `P0-HOOK` — this ensures it is fetched and verified before other MGs in step 4

#### D. Assign Segment Patterns

Based on the conceptual storyboard's `visual_type` column:
- `hook-text` → **DEPRECATED as opener.** Use V5 (split-screen, 50/50, 1-2s) as the hook instead. V6 may be used mid-video for emphasis only. **The first V5 segment's overlay MUST be the hero keyword visual from C2. Text MGs are forbidden for this segment.**
- `speaker` → V1 (SubtleZoom, 1.03 zoom — per production style guide)
- `speaker-zoom` → V2 (SubtleZoom, 1.08 zoom — per production style guide)
- `broll` → V5 (VerticalSplitScreen with MG overlay) or V4 (full-screen MG). **Never V3** — all beats marked 'broll' or 'screen-recording' MUST be assigned V5 (split-screen with MG overlay) or V4 (full-screen MG).
- `motion-graphic` → V4 (MotionGraphic)
- `motion-graphic-cutaway` → V4b (MotionGraphic, no caption, 60-90 frames)
- `split-screen` → V5 (VerticalSplitScreen). **V5 overlay safe margin:** The 16:9 overlay displayed in the ~9:8 top zone loses ~16% from each side via `objectFit: cover`. Mitigation:
  - **Prompt-level:** When writing Hera prompts for V5 overlay MGs, instruct key content to be centered with 15% safe margin on left/right edges.
  - **Remotion padding (sole mechanism):** The `overlayPadding` prop in VerticalSplitScreen renders the MG at 85% size with `objectFit: contain` and `#F5F5F5` background. This is the only padding applied — FFmpeg pre-padding (`v5Padding`) is deprecated and no longer used during upscaling.
- `cta` → V7 (SubtleZoom 1.05 + SocialProofStack)

**Note:** `layoutContext` is no longer needed for caption positioning (captions are platform-native). It may still be recorded for future reference but has no functional impact on Remotion codegen.

#### D2. MG Cutaway Interleaving

After initial pattern assignment, scan the timeline and insert MG cutaways to meet pacing requirements:

1. **Speaker segment >3s → split with MG cutaway:** Any speaker segment (V1/V2) longer than 3 seconds (90 frames @ 30fps) should be split. Insert a 2-3s V4b MG cutaway in the middle, creating two shorter speaker segments flanking the cutaway.

2. **Gap >10s without V4/V4b → insert MG cutaway:** Scan the timeline for any stretch of 10+ seconds without a motion graphic segment. Insert a contextual V4b cutaway to break the monotony.

3. **MG content selection rules:**
   - Tool/platform mentioned in voiceover → tool logo MG (e.g., Claude, Node.js, VS Code)
   - Build/creation action described → typing/coding animation
   - Statistic or number cited → animated counter or data visualisation
   - Comparison being made → side-by-side graphic
   - Default → abstract tech/data flow
   - **Note:** The hook MG (P0-HOOK from C2) is already assigned to the first V5 segment. Do not re-use the same hero keyword visual as a cutaway unless the tool is explicitly mentioned again later in the script.

4. **Mandatory zoom rule:** Every V1/V2/V7 segment that follows a non-speaker segment (V3, V4, V4b, V5, V6) must use SubtleZoom — no exceptions. This creates the "re-entry" energy that top creators use on every speaker return.

#### D3b. Visual Description Brief — Mandatory Prompt Format for ALL Tier C MGs

**Every Tier C MG prompt MUST use the 5-section Visual Description Brief (VDB) format below. Generic prompts like "Claude Code terminal with command text animating" are a SYSTEM FAILURE and will be rejected by the step-04 quality gate.**

**The #1 goal: INTERFACE FIDELITY.** Competitor MGs look like the REAL tool interfaces — not generic dashboards or code editors. Our reference images give Hera the visual target, but Hera cannot interpret a reference image well without detailed text guidance. The prompt must redundantly describe what's in the reference image so Hera has BOTH visual and textual signals. The more the prompt describes the actual interface structure, the more realistic the output.

**CRITICAL CONSTRAINTS:**
- **NEVER ask Hera to draw/generate a logo** — logos always render incorrectly. If an MG needs a logo, provide it as a reference image via `fetch-logo.ts` with fidelity instruction. Prefer Tier A (Remotion-native) for logo animations.
- **Minimise text in MGs** — only include text that's critical to understanding and transcript sync. Too much text causes rendering quality issues. Max 4-5 distinct text strings per MG.
- **Multi-tool MGs** — use text labels with brand colours, NOT multiple logo images. Hera garbles multi-logo composites.
- **Reference images are mandatory for all tool-UI Tier C MGs** — library or frame-extract. Clean reference images: remove speaker PiP, consider clearing irrelevant text that Hera might reproduce.

##### VDB Section A — Visual Inventory & UI Anatomy (Interface Fidelity)

The MG must look like the REAL tool interface. Describe the UI as if Hera has never seen this tool before, because it hasn't:

- **Name the tool**, then describe its ACTUAL layout from the reference image: "VS Code has a narrow vertical icon sidebar on the far left (dark grey, ~48px wide), a file explorer panel (250px, slightly lighter grey), and a main editor area. The bottom 30% is an integrated terminal with a dark background."
- **Describe the interface's signature visual elements** — the things that make it immediately recognisable: VS Code's blue status bar at bottom, Claude's orange-on-white chat bubbles, YouTube Studio's red accent buttons, n8n's canvas with connected node cards.
- **Every UI component relevant to the animation**: sidebars, panels, buttons, input fields, navbars, modals — with their relative positions and sizes.
- **Exact text that should appear** — but ONLY critical text that supports what the viewer is hearing. Strip unnecessary labels, menu items, or placeholder text. Max 4-5 distinct text strings.
- **Images/icons**: what they depict (but NOT logos — never ask Hera to generate a logo).
- The reference image + this description together should make it impossible for Hera to produce a generic-looking UI.

##### VDB Section B — Typography & Colour (2+ hex codes minimum)

- Background colour (hex), surface/card colours, text colours, accent colours — minimum 2 hex codes.
- Font weight and family description for each text element (e.g., "monospace, medium weight" for terminal text).
- Text size relative to canvas (e.g., "~14pt equivalent", "large heading ~32pt").
- Text effects if any: shadow, outline, glow, background pill.
- **CRITICAL: Text economy.** Only include text that directly supports what the viewer is hearing. Remove decorative or irrelevant UI text. If showing a terminal: 2-3 key command lines, not full history. If showing a dashboard: headline + 1-2 key data points, not every label.
- style suffix: light outer background (#F5F5F5), dark text, green accents (#2D6A4F).

##### VDB Section C — Motion Choreography (numbered steps with timing)

- Entry → active phase → exit, step-by-step with approximate timing.
- For UI mockups: describe cursor movements, typing sequences (character by character?), click targets, hover states, and how the UI responds to each interaction.
- Zoom/pan: specify exact start% → end% and direction (e.g., "slow zoom 100% → 106% centred on terminal panel").
- State changes: empty → loading → populated, with timing.
- Minimum 3 steps for 2-3s MGs, 5 steps for 3-4s MGs.

**Zoom-to-focal-content strategy (STRONGLY RECOMMENDED for IDE/editor/terminal MGs):**
IDE and tool interface frames are visually busy. Direct Hera to zoom into the focal area where the key action happens — this naturally eliminates surrounding UI noise.

**Zoom patterns:**
- **Type-and-zoom:** Start wide (matching reference), zoom into typing/logging area. Best for terminal commands, chat inputs.
- **Zoom-to-result:** Start zoomed on action area, data/text appears. Best for dashboards, reports.
- **Static zoom:** Already zoomed into focal area for entire duration. Best for short 2s MGs.

##### VDB Section D — Transcript Sync (MANDATORY — null/N/A is a SYSTEM FAILURE)

- **Quote the EXACT transcript words spoken during this segment** — calculate the time window from startFrame/durationInFrames, find the matching transcript, quote it verbatim.
- **Describe how each visual element reinforces those specific words.** E.g., speaker says "scans 10 YouTube channels" → MG shows progress bar filling with counter 0→10 and text "Scanning channels...".
- The MG should directly illustrate what the viewer is hearing. This is what makes competitor MGs feel engaging — every visual is perfectly synchronised to the narration.

##### VDB Section E — Reference Image & Composition

- **If reference image provided** — describe what IS in the reference image in extreme detail: the layout structure, colour scheme, and specific content visible. Think of it as: the reference image shows Hera WHAT the interface looks like, and the prompt tells Hera WHICH parts to focus on, HOW to animate them, and WHAT specific content to display.
- **Logo fidelity** — if reference image is a logo: "The reference image contains the exact logo to use. Reproduce this logo precisely as shown — do not redraw, reinterpret, or stylise it."
- Layout structure, safe margins for V5 overlays (15% left/right).
- Exclusions: "Do not include any people or faces."
